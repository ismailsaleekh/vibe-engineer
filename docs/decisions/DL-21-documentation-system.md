## Decision summary

The documentation system for `vibe-engineer` is repository-local Markdown as the source of truth, rendered later as a static public documentation site using VitePress, with generated references produced from actual typed/package/CLI/schema/API sources. The documentation IA is fixed now; implementation, site config, generated references, docs tests, examples, and CI wiring are future work owned by `I-24-docs-reference-governance-polish` or the sibling implementation lanes named below.

Documentation closure is evidence-bound. Any page, snippet, command, API reference, generated starter walkthrough, governance link, security note, or observability note that claims live behavior must be proven later against the actual binary/package/schema/site/tool output. Hand-maintained references that can drift from source are not authoritative.

## Decision details

1. The canonical authoring format is Markdown/MDX-compatible Markdown stored in the harness repo under `docs/**`, except generated references whose source of truth is the actual typed source, schema, CLI, or contract that generated them.
2. The public docs renderer is VitePress, selected for a static, Markdown-first, TypeScript-friendly docs site with a small configuration surface. `I-24` must implement or split a public-docs sublane for `docs/.vitepress/**` and any docs build scripts before claiming public-site proof.
3. Docs content is structured by stable information architecture and typed source-of-truth rules, not by ad hoc prose. Every generated/reference page must say what source produced it and what freshness check proves it is current.
4. DL-21 does not implement docs, site tooling, reference generators, examples, tests, CI, root config, package source, or starter files.
5. Sibling-owned content remains sibling-owned. DL-21 defines documentation slots and gates; it does not invent governance policy (`DL-19`), security policy (`DL-22`), observability policy (`DL-23` until green), exact CLI/schema/API internals (`DL-02`, `DL-07`, `DL-10`, `DL-14`), starter architecture (`DL-16`), context bootstrap (`DL-17`), CI behavior (`DL-18`), or implementation facts.
6. Documentation must preserve locked decisions: `vibe-engineer`, two-repo direction, domain-neutral core, fixed starter stack, six skills, artifact flow, automatic verification/context/evidence, deterministic blockers/advisory boundary, mechanical gates, and no push/PR without approval.

## Alternatives considered

### Repository-local Markdown only

- decision: rejected as sufficient; accepted as source format.
- rationale: source docs must be reviewable in the repo, but backlog §21 also requires public docs/tool direction and generated-reference freshness. Markdown alone does not give site build/link checks or public navigation.
- consequences: Markdown remains the authoring source; VitePress renders it and docs-quality gates verify it.

### Full public docs site immediately

- decision: rejected for DL-21 execution scope.
- rationale: this decision-only lane owns no docs implementation paths, site config, package scripts, CI, or generated references. Implementing the site now would violate ownership.
- consequences: VitePress is selected now; implementation waits for `I-24` with explicit path ownership and live build/link evidence.

### Generated API/reference only with no narrative docs

- decision: rejected.
- rationale: source docs require skills, workflow, architecture, contribution, migration/adoption, agent-building, evidence, context, and dirty-tree discipline. Generated references alone do not teach how to use or extend the harness.
- consequences: generated references are mandatory but live alongside narrative guides and architecture docs.

### Hand-maintained API/CLI/schema reference

- decision: rejected as authoritative.
- rationale: hand-maintained references drift from actual JSON Schemas, CLI envelopes, package exports, and API contracts. DL-02/DL-07/DL-10/DL-14 require typed contracts and evidence.
- consequences: hand-authored explanation may wrap generated references, but canonical reference tables must be generated or freshness-validated from actual sources.

### Selected approach: Markdown + VitePress + generated references + doc-tested examples

- decision: accepted.
- rationale: gives stable repo-local docs, a public site path, deterministic build/link/docs-test gates, and source-derived references while preserving sibling boundaries.
- consequences: `I-24` can implement public docs without inventing IA/tooling; generated/live seams stay pending until actual providers exist.

## API/reference generation policy

All reference docs that can drift must be generated or freshness-checked from actual source-of-truth artifacts.

Required source-of-truth map:

- Artifact schemas and manifests: canonical JSON Schemas and generated TypeScript types from `packages/artifacts` / `DL-02` implementation outputs.
- CLI commands/options/status/exit/envelope: actual `vibe-engineer` binary metadata/help and machine result schemas from `DL-07`/I-02/I-09/etc.; never copied prose or screenshots.
- Package public exports: TypeDoc or equivalent TypeScript API extraction over built package source and public export maps.
- API contracts/OpenAPI/client: generated from `ts-rest` + named Zod contract source selected by `DL-14`; OpenAPI is a generated projection only.
- Verification/evidence/failure taxonomy: generated or validated from `DL-10`/I-09 runner/catalog/evidence sources.
- Governance/license/security/versioning references: links to actual root files from `I-00` plus `DL-19`/`DL-22` policy.

Stale-reference failure policy:

- Generated reference output must carry source path/version/hash or equivalent provenance where feasible.
- A docs freshness check must fail if generated reference output is missing, stale relative to source, hand-edited as canonical, or not reproducible.
- If actual source does not exist yet, the affected reference page is a placeholder with explicit blocker and owner; it must not claim current behavior.
- Hand-authored narrative can explain generated reference but cannot override it.

Later proof owner: `I-24` coordinates docs generation/freshness; source implementation lanes own the real producer sources; validators must inspect actual generated output and generation evidence.

## Examples and tutorials policy

Examples and tutorials must be domain-neutral, executable when they claim execution, and clearly classified.

Rules:

1. Core examples demonstrate harness workflow and structure, not a business domain.
2. Any sample/demo/reference starter content must be labeled `sample`, `demo`, or `reference`, isolated from core policy, and replaceable by consuming projects.
3. Tutorials that run commands must be doc-tested against the actual `vibe-engineer` binary in non-interactive machine mode once the binary exists.
4. API/package snippets must typecheck or execute against actual package/API exports once those exports exist.
5. CLI snippets must assert result-envelope status/exit behavior, not scrape prose.
6. Generated starter examples must prove the starter consumes harness packages rather than copying harness logic.
7. Snippets that cannot be executed in the current environment must be marked pending-live/BLOCKED for I-24 closure unless explicitly non-executable conceptual prose.
8. Examples must include negative or failure-path docs where safety/verification claims are made, such as invalid config, stale reference, missing evidence, rejected secret, failed docs test, or blocked command.

## Architecture, migration, contribution, and agent-building docs policy

- repo/package architecture;
- two-repo starter/harness relationship;
- skill/orchestrator/artifact flow;
- verification/evidence/context architecture;

### Migration/adoption/upgrade guides

Migration/adoption docs must include:

- creating/importing a project using the actual CLI;
- adopting the harness in an existing repo when import exists;
- upgrading `vibe-engineer` versions using the actual update/check/plan behavior when implemented;

Docs must link to actual `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`, `CHANGELOG.md`, license, and API-stability policy from `I-00`/DL-19. DL-21 does not define contribution governance. It defines docs placement, discoverability, link validation, and stale-link checks only.

### How to build with agents

Agent-building docs must cover:

- six-skill workflow: `brainstorm`, `grill-me`, `task`, `plan`, `build`, `ship`;
- raw intent → Work Brief → Implementation Plan with Verification Delta → Build Result → Ship Packet;
- agent registry and validation at a conceptual level, with exact registry schemas from DL-02/DL-05;
- selected harness adapter docs from DL-06/I-14 when implemented;
- Triad discipline and no self-validation;
